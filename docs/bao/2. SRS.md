# PHẦN 1: GIỚI THIỆU

## 1.1. Mục tiêu dự án

Xây dựng một **chat bot hỏi đáp dựa trên tài liệu môn học** (documents-based QA) cho phép sinh viên đặt câu hỏi về nội dung học phần và nhận câu trả lời được trích dẫn nguồn từ tài liệu đã được upload. Đồng thời **nghiên cứu và so sánh hiệu quả** giữa hai phương pháp **RAG (Retrieval-Augmented Generation)** và **fine-tuning** trong bối cảnh tiếng Việt.

### Mục tiêu kỹ thuật
- Triển khai kiến trúc 3-layer (DAL/BLL/Presentation) bằng C# (.NET 8+).
- Sử dụng SQL Server làm database chính, kết hợp vector store (FAISS/Chroma/Qdrant) cho embedding.
- Xây dựng pipeline ingestion: OCR (nếu PDF scan) → extract text → chunk → embed → lưu vector.
- Xây dựng RAG flow: query rewrite (tùy chọn) → retrieve (top-k) → rerank (tùy chọn) → generate + citations.
- Cung cấp dashboard nghiên cứu (RBL) để benchmark các embedding model và chunking strategy.
- Đảm bảo tính truy vết: mọi câu trả lời phải kèm citation (file + trang/slide + chunk id).
- Đảm bảo bảo toàn phạm vi kiến thức: chỉ trả lời dựa trên tài liệu đã index.

### Mục tiêu nghiên cứu
- So sánh RAG (various embeddings + chunking) với fine-tuned model.
- Benchmark các metric RAGAS: Faithfulness, Answer Relevance, Context Recall.
- Phân tích kết quả theo độ khó câu hỏi (Easy/Medium/Hard).

## 1.2. Phạm vi

### Trong phạm vi (In-Scope)
- Upload và index tài liệu: PDF, DOCX, slide bài giảng.
- Quản lý tài liệu theo môn học / chương (demo 1 môn).
- Chat đa phiên với lịch sử hội thoại.
- Tích hợp RAG với các embedding model đã định nghĩa.
- Dashboard nghiên cứu với bảng tổng hợp, lọc theo embedding model/chunking/top-k/prompt.
- Logging chi tiết cho từng request (retrieved chunks, score, câu trả lời, citations, latency, cost).
- Export kết quả benchmark (RAGAS metrics).

### Ngoài phạm vi (Out-of-Scope)
- OCR phức tạp (chỉ dùng fallback đơn giản).
- Fine-tuning thực tế: code sẽ mô phỏng interface nhưng demo trọng RAG; fine-tuning nếu có sẽ là demo nhỏ.
- Multi-tenancy, role-based auth phức tạp (chỉ cần demo auth đơn giản).
- Scale production: tập trung vào tính đúng đắn và benchmark.

## 1.4. Từ viết tắt (Definitions & Acronyms)

| Thuật ngữ | Giải thích |
|-----------|------------|
| RAG | Retrieval-Augmented Generation: kỹ thuật lấy context từ kho tài liệu rồi đưa vào LLM để trả lời |
| Embedding | Vector biểu diễn văn bản dùng để tìm kiếm tương đồng |
| Chunk | Đoạn văn bản nhỏ sau khi tách tài liệu |
| Citation | Trích dẫn nguồn: tên file + số trang/slide + chunk id |
| RAGAS | RAG Assessment framework: Faithfulness, Answer Relevance, Context Recall, Context Precision |
| Top-k | Số lượng chunk được retrieve cho mỗi query |
| Rerank | Bước tái sắp xếp các chunk retrieved để tăng độ chính xác |
| OCR | Optical Character Recognition: trích xuất text từ PDF scan |

# PHẦN 2: MÔ TẢ TỔNG QUAN HỆ THỐNG

## 2.1. Kiến trúc tổng thể

```
┌─────────────────────────────────────────────────────────────┐
│                     Presentation Layer                      │
│    (ASP.NET Core Web App: MVC + Razor Pages Dashboard)      │ 
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                     Business Logic Layer                    │
│  Services: Chat, Document, Experiment, Benchmark, RAG       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                     Data Access Layer                       │
│  Repositories: SQL Server + Vector Store (FAISS/Chroma)     │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    External Services                        │
│  Embedding API (OpenAI/PhoBERT/e5)  │  LLM API (GPT/Claude) │
└─────────────────────────────────────────────────────────────┘
```

- **Presentation Layer**: ASP.NET Core Web Application sử dụng kết hợp MVC và Razor Pages cho các module khác nhau.
- **Business Logic Layer (BLL)**: Các Service xử lý nghiệp vụ: DocumentService, ChatService, ExperimentService, RAGService.
- **Data Access Layer (DAL)**: Repository kết nối SQL Server lưu metadata, chat history, experiment logs; và VectorStoreAdapter kết nối FAISS/Chroma/Qdrant lưu embeddings.
- **External Services**: Gọi embedding model API (có thể local) và LLM API (OpenAI/Anthropic).

## 2.2. Module chính

1. **Document Management Module**: Upload, index, re-index, xem danh sách và chi tiết tài liệu.
2. **Chat Module**: Tạo phiên chat, gửi câu hỏi, nhận câu trả lời có citations, lưu lịch sử.
3. **Experiment Module (RBL)**: Chạy benchmark với cấu hình khác nhau, lưu logs, hiển thị dashboard so sánh.
4. **Admin/Configuration Module**: Quản lý môn học, chương, cấu hình hệ thống.

# PHẦN 3: YÊU CẦU CHỨC NĂNG (FUNCTIONAL REQUIREMENTS)

## 3.1. Module Quản lý Tài liệu (Document Management)

### FR-DOC-01: Upload tài liệu
- **Mô tả**: Người dùng upload file (PDF, DOCX, PPT, PPTX) cho một môn học và chương.
- **Đầu vào**: File, CourseId, ChapterId (tùy chọn).
- **Xử lý**:
  1. Validate file type và size (max 50MB).
  2. Lưu file vào storage (local filesystem hoặc cloud).
  3. Tạo bản ghi Document trong DB với trạng thái `Pending`.
  4. Enqueue job indexing (background queue).
- **Đầu ra**: DocumentId, trạng thái ban đầu `Pending`.
- **Constraint**: Upload đồng thời nhiều file được hỗ trợ.

### FR-DOC-02: Index tài liệu (background job)
- **Mô tả**: Xử lý tài liệu đã upload: extract text → chunk → embed → lưu vector.
- **Xử lý**:
  1. Đọc file, extract text (PDF: iTextSharp/MuPDF; DOCX: OpenXML; PPT: Aspose).
  2. Nếu PDF là scan, dùng OCR fallback (Tesseract).
  3. Chunk văn bản theo cấu hình: fixed-size (300/500/800 tokens) hoặc structure-based (theo heading).
  4. Gọi embedding model API để chuyển chunk thành vector.
  5. Lưu vector vào Vector Store với metadata (documentId, chunkId, pageNumber, text, embeddingModel, chunkingStrategy).
  6. Cập nhật trạng thái Document thành `Indexed` hoặc `Failed`.
- **Output**: Log số chunk đã index.

### FR-DOC-03: Re-index tài liệu
- **Mô tả**: Chạy lại pipeline index với cấu hình mới (thay đổi embedding model/chunking strategy).
- **API**: `POST /documents/{id}/reindex` với body { embeddingModel, chunkingStrategy }.

### FR-DOC-04: Xem danh sách và chi tiết tài liệu
- **Mô tả**: Hiển thị bảng danh sách tài liệu với thông tin: tên, loại, môn học, thời gian upload, trạng thái index.
- **Chi tiết**: Hiển thị metadata, số chunk, cấu hình index, nút re-index/xóa.

### FR-DOC-05: Xóa tài liệu khỏi index
- **Mô tả**: Xóa bản ghi Document và toàn bộ chunk liên quan trong DB và Vector Store.
- **Constraint**: Có cảnh báo xác nhận.

## 3.2. Module Chat & Hỏi đáp

### FR-CHAT-01: Tạo phiên chat
- **API**: `POST /chat/sessions` với body { CourseId, Title (tùy chọn) }
- **Output**: SessionId.

### FR-CHAT-02: Gửi câu hỏi và nhận câu trả lời
- **API**: `POST /chat/sessions/{id}/messages`
- **Input**: { Content, TopK (tùy chọn), UseRerank (tùy chọn) }
- **Pipeline RAG**:
  1. Query rewrite (optional): diễn đạt lại câu hỏi cho rõ ràng hơn.
  2. Retrieve: Gọi VectorStore.search(queryEmbedding, topK) để lấy top-K chunk.
  3. Rerank (optional): Dùng cross-encoder rerank các chunk để chọn top-N.
  4. Build prompt: Context (các chunk), chat history, system prompt (rằng chỉ trả lời dựa trên context).
  5. Gọi LLM API để sinh câu trả lời.
  6. Parse citations từ metadata của các chunk được sử dụng.
  7. Lưu ChatMessage (question + answer + citations) vào DB.
- **Guardrail**:
  - Nếu không tìm thấy chunk có similarity score trên threshold (ví dụ 0.6), trả về message "Không tìm thấy thông tin liên quan trong tài liệu."
  - Nếu LLM trả lời ngoài context, có thể dùng RAGAS để detect.
- **Output**: { Answer, Citations: [{ DocumentName, PageNumber, ChunkId, Text }], Latencies: { retrieve, generate, total } }

### FR-CHAT-03: Lấy lịch sử phiên chat
- **API**: `GET /chat/sessions/{id}/messages` (pagination support).
- **Output**: Danh sách message theo thứ tự thời gian.

## 3.3. Module Nghiên cứu (RBL - Research & Benchmark)

### FR-EXP-01: Chạy thử nghiệm benchmark
- **API**: `POST /experiments/run`
- **Input**:
```json
{
  "Config": {
    "EmbeddingModel": "multilingual-e5-base | text-embedding-3-small | PhoBERT-base | bge-m3",
    "ChunkingStrategy": "fixed-size | structure-based | hybrid",
    "ChunkSize": 300/500/800,
    "ChunkOverlap": 0.1,
    "TopK": 3/5/8,
    "RerankEnabled": true/false,
    "PromptTemplateId": "default | enhanced",
    "LlmModel": "gpt-4o-mini | claude-3-5-sonnet"
  },
  "TestSetId": "id của test set 50 câu hỏi đã chuẩn bị"
}
```
- **Xử lý**:
  1. Tạo ExperimentRun với trạng thái `Running`.
  2. Với mỗi câu hỏi trong test set:
     - Thực thi pipeline RAG theo config.
     - Lưu kết quả: retrieved chunks, similarity scores, câu trả lời, citations, latency, cost.
  3. Tính RAGAS metrics: Faithfulness, Answer Relevance, Context Recall (so với ground truth).
  4. Lưu tổng hợp metrics vào ExperimentRun.
  5. Cập nhật trạng thái `Completed`.

### FR-EXP-02: Xem danh sách và kết quả thử nghiệm
- **API**:
  - `GET /experiments` (lọc theo embeddingModel, chunkingStrategy, ngày).
  - `GET /experiments/{id}` (chi tiết: cấu hình, metrics tổng hợp, breakdown theo độ khó câu hỏi).
- **Dashboard UI**:
  - Bảng tổng hợp: Config → RAGAS scores, latency, cost.
  - Lọc: theo embedding model / chunking strategy.
  - Biểu đồ: so sánh metrics giữa các cấu hình.

### FR-EXP-03: Quản lý Test Set
- **Upload test set CSV**: cột: Question, GroundTruthAnswer, Difficulty (easy/medium/hard), Chapter.
- **Export test set** (mẫu).

# PHẦN 4: YÊU CẦU PHI CHỨC NĂNG (Non-Functional Requirements)

### NFR-01: Bảo toàn phạm vi kiến thức (Grounding)
- Mọi câu trả lời phải được ràng buộc trong context được retrieve.
- System prompt bắt buộc: "Chỉ trả lời dựa trên tài liệu đã cung cấp. Nếu không có thông tin, hãy nói 'Không tìm thấy thông tin trong tài liệu.'"
- Tích hợp detection khi LLM hallucinate (so sánh với source chunks).

### NFR-02: Tính truy vết (Traceability)
- Mỗi câu trả lời phải có danh sách citations với ít nhất: DocumentName, PageNumber/SlideNumber, ChunkId.
- UI hiển thị citation với link trỏ đến vị trí trong tài liệu (nếu có viewer).

### NFR-03: Hiệu năng (Performance)
- Latency target: retrieve < 200ms, generate < 2s (tùy LLM).
- Logging chi tiết theo từng stage để benchmark.
- Support concurrent chat sessions (ít nhất 10 simultaneous).

### NFR-04: Khả năng tái lập (Reproducibility)
- Mỗi ExperimentRun lưu đầy đủ: config (embedding, chunking, top-k, prompt, LLM), version code (git commit), version test set.
- Đảm bảo cùng config với cùng test set cho ra kết quả giống nhau (deterministic embedding nếu có seed).

### NFR-05: Khả năng quan sát (Observability)
- Logging structured (JSON) với các fields: timestamp, requestId, stage, latency, embeddingModel, chunkingStrategy, topK, numRetrieved, promptTokens, completionTokens, cost.
- Dashboard có thể filter và export logs.

### NFR-06: Bảo mật cơ bản
- Simple authentication (Identity Framework) cho demo.
- Rate limiting cho API chat (tránh abuse).
- Validate upload file (chỉ cho phép extensions an toàn).

# PHẦN 5: MÔ HÌNH DỮ LIỆU (Conceptual Data Model)

## 5.1. Entities

### Course (Môn học)
- CourseId (PK)
- Name, Code, Description
- CreatedAt, UpdatedAt

### Chapter (Chương/Mục) - Optional
- ChapterId (PK)
- CourseId (FK)
- Title, OrderNumber
- CreatedAt, UpdatedAt

### Document (Tài liệu)
- DocumentId (PK)
- CourseId (FK), ChapterId (FK nullable)
- OriginalFileName, StoredFilePath
- FileType (PDF/DOCX/PPT/PPTX)
- FileSize
- UploadedBy (UserId), UploadedAt
- IndexStatus: Enum (Pending, Processing, Indexed, Failed)
- IndexErrorMessage (nullable)
- Metadata (JSON: pageCount, author, etc.)
- CreatedAt, UpdatedAt

### Chunk (Đoạn văn bản đã index)
- ChunkId (PK)
- DocumentId (FK)
- ChunkIndex (vị trí trong document)
- PageNumber (nullable), SlideNumber (nullable)
- Text (nvarchar(max))
- EmbeddingVectorId (PK/foreign key vào Vector Store record)
- EmbeddingModel (string)
- ChunkingStrategy (string)
- CreatedAt

### ChatSession (Phiên chat)
- SessionId (PK)
- UserId (FK), CourseId (FK)
- Title, CreatedAt, LastActivityAt

### ChatMessage (Tin nhắn)
- MessageId (PK)
- SessionId (FK)
- Role (User/Assistant/System)
- Content (nvarchar(max))
- Citations (JSON array: [{ DocumentId, PageNumber, ChunkId, Snippet }])
- PromptTokens, CompletionTokens, LatencyMs
- CreatedAt

### ExperimentRun (Thử nghiệm benchmark)
- RunId (PK)
- Name, Description
- Config (JSON: { embeddingModel, chunkingStrategy, chunkSize, chunkOverlap, topK, rerankEnabled, promptTemplateId, llmModel })
- TestSetId (FK)
- Status: Enum (Running, Completed, Failed)
- StartedAt, CompletedAt
- Metrics (JSON: { faithfulness, answerRelevance, contextRecall, avgLatency, totalCost })
- GitCommitHash, CodeVersion (nullable)
- ErrorMessage (nullable)

### TestSet (Tập câu hỏi benchmark)
- TestSetId (PK)
- Name, Description
- OwnerId, CreatedAt
- Questions (JSON hoặc bảng TestQuestion riêng)

### TestQuestion (Chi tiết câu hỏi)
- QuestionId (PK), TestSetId (FK)
- QuestionText, GroundTruthAnswer
- Difficulty (Easy/Medium/Hard)
- Chapter (optional để phân tích coverage)

# PHẦN 7: ASSUMPTIONS, CONSTRAINTS & RISKS

## 7.1. Assumptions
- Có kết nối internet để gọi external embedding/LLM APIs (nếu dùng OpenAI).
- Tài liệu đầu vào có cấu trúc tương đối rõ ràng (có heading nếu muốn structure-based chunking).
- Embedding models đã được chọn từ danh sách: multilingual-e5-base, text-embedding-3-small, PhoBERT-base, bge-m3.
- LLM backend có sẵn API (OpenAI, Anthropic, hoặc local model).

## 7.2. Constraints
- Database chính: SQL Server.
- Backend: C# (.NET 8+).
- Kiến trúc: 3-layer (DAL/BLL/Presentation).
- Tài liệu chỉ hỗ trợ PDF/DOCX/PPT/PPTX.
- Benchmark tập trung vào RAG; fine-tuning chỉ ở mức interface/demo.

## 7.3. Risks & Mitigations

| Risk | Mức độ | Mitigation |
|------|--------|------------|
| OCR kém chất lượng → text trích xuất không đầy đủ | Medium | Dùng fallback OCR tốt (Tesseract 5), nếu fail thì báo lỗi rõ ràng, user có thể upload lại |
| Embedding model không hỗ trợ tiếng Việt tốt | High | Chỉ benchmark các model đã được khuyến nghị cho tiếng Việt; có pre-check embedding quality |
| LLM sinh câu trả lời hallucinate ngoài context | High | System prompt mạnh, detection logic, RAGAS faithfulness metric |
| Vector store không scale với số lượng chunk lớn | Low | Demo nên dùng dataset nhỏ (1 môn học); Qdrant nếu cần scale |
| Latency cao do nhiều API call | Medium | Async pipeline, caching embedding cho chunk cố định, tối ưu top-k |

# PHẦN 8: ACCEPTANCE CRITERIA

## 8.1. Tổng thể
- Hệ thống chạy được end-to-end: upload → index → chat với citations.
- Có dashboard benchmark với ít nhất 3 cấu hình khác nhau.
- Code sạch, theo 3-layer, có unit test cơ bản cho BLL/DAL.

## 8.2. Chức năng
- Upload file đúng type, trạng thái chuyển từ Pending → Indexed.
- Chat trả về câu trả lời có citations; nếu không có context, báo "Không tìm thấy thông tin trong tài liệu."
- Lịch sử chat được lưu và hiển thị.
- ExperimentRun chạy hoàn tất và lưu được RAGAS metrics.

## 8.3. Phi chức năng
- Latency retrieve < 500ms; generate < 5s (lưu ý tùy LLM).
- Logging đầy đủ để trace từng request.
- Bảo mật cơ bản: login, authorize theo user.
