# Solution Guide - Triển khai dự án theo kiến trúc 3-Layer với ASP.NET Core

## 1. Mục tiêu tài liệu

Tài liệu này hướng dẫn cách triển khai dự án **chat-bot hỏi đáp dựa trên tài liệu môn học** bằng **C# / ASP.NET Core**, sử dụng **kiến trúc 3-layer** và **SQL Server** làm cơ sở dữ liệu chính. Ngoài ra, tài liệu cũng **so sánh ASP.NET Core MVC và Razor Pages** để giúp nhóm lựa chọn cách tổ chức giao diện phù hợp cho từng module.

---

## 2. Công nghệ đề xuất

### Back-end / Framework
- .NET 8 hoặc mới hơn
- ASP.NET Core Web App
- Entity Framework Core
- SQL Server
- Dependency Injection built-in của ASP.NET Core
- Hosted Service / Background Service cho indexing và benchmark

### AI / Search / Storage
- Vector Store: Qdrant / Chroma / FAISS (khuyên dùng Qdrant nếu tách service)
- Embedding models: multilingual-e5-base, PhoBERT-base, bge-m3, text-embedding-3-small
- LLM API: OpenAI / Anthropic / local model server
- File storage: Local file system hoặc Azure Blob / S3 (nếu triển khai cloud)

### Logging / Monitoring
- Serilog
- Seq / ELK / Application Insights (tùy môi trường)

## 3. Kiến trúc 3-Layer áp dụng cho dự án

### 3.1. Tổng quan

Kiến trúc 3-layer được chia như sau:

1. **Presentation Layer**
   - Giao diện web cho upload tài liệu, chat, dashboard benchmark
   - MVC Controllers / Razor Pages / API endpoints

2. **Business Logic Layer (BLL)**
   - Xử lý nghiệp vụ chính: tài liệu, chunking, embedding, retrieval, chat, benchmark
   - Điều phối pipeline và validate rule nghiệp vụ

3. **Data Access Layer (DAL)**
   - Truy cập SQL Server
   - Tương tác vector store
   - Lưu/đọc file metadata, chat history, experiment logs

### 3.2. Luồng xử lý tiêu biểu

#### Luồng upload tài liệu
Presentation → DocumentController/PageModel → DocumentService → DocumentRepository → SQL Server

Sau đó background worker xử lý:
BackgroundWorker → FileProcessingService → ChunkingService → EmbeddingService → VectorStoreService

#### Luồng chat
Presentation → ChatController/PageModel → ChatService → RetrievalService → LLMService → ChatRepository → Response UI

#### Luồng benchmark
Presentation → ExperimentController/PageModel → ExperimentService → RAGPipelineService → MetricsService → SQL Server / Dashboard UI

## 5. So sánh ASP.NET Core MVC và Razor Pages

## 5.1. MVC là gì?

MVC chia ứng dụng theo:
- **Model**: dữ liệu / business model
- **View**: giao diện hiển thị
- **Controller**: nhận request, xử lý flow, trả response

### Ưu điểm MVC
- Phù hợp hệ thống có nhiều action, nhiều API endpoint
- Routing rõ ràng
- Dễ mở rộng cho admin dashboard và module phức tạp
- Hợp với kiến trúc quen thuộc trong đồ án doanh nghiệp
- Tốt khi muốn tách web UI và API style rõ ràng

### Nhược điểm MVC
- Với page CRUD đơn giản có thể tạo nhiều file hơn mức cần thiết
- Controller dễ phình to nếu tổ chức không tốt

---

## 5.2. Razor Pages là gì?

Razor Pages tổ chức theo **trang**, mỗi trang có:
- `.cshtml`
- `.cshtml.cs` (PageModel)

Mỗi page chịu trách nhiệm cho một màn hình cụ thể.

### Ưu điểm Razor Pages
- Dễ phát triển nhanh các trang CRUD
- Tổ chức theo page trực quan
- Ít ceremony hơn MVC cho các màn hình quản trị nội bộ
- Phù hợp các màn hình như Documents List, Document Detail, Experiment Detail

### Nhược điểm Razor Pages
- Không trực quan bằng MVC khi hệ thống có nhiều luồng nghiệp vụ phân tán theo action
- Với app có nhiều API và flow phức tạp, tổ chức dài hạn có thể kém rõ ràng hơn MVC

---

## 5.3. Nên chọn cái nào cho dự án này?

### Khuyến nghị thực tế cho dự án
**Nên dùng kết hợp cả hai**:

- **MVC** cho:
  - Chat module
  - API endpoints
  - Ajax/JSON responses
  - Webhook/background task trigger
  - Tương tác AI pipeline phức tạp

- **Razor Pages** cho:
  - Document management pages
  - Experiment dashboard pages
  - Course/chapter CRUD
  - Các màn hình quản trị đơn giản

### Kết luận lựa chọn
Nếu nhóm muốn:
- **Dễ trình bày kiến trúc, dễ mở rộng, chuẩn enterprise hơn** → ưu tiên **MVC làm xương sống**.
- **Làm nhanh giao diện CRUD và ít boilerplate** → dùng **Razor Pages** cho phần dashboard/quản trị.

**Đề xuất tốt nhất cho đồ án này:**
> Dùng **ASP.NET Core MVC làm nền chính**, và có thể tổ chức một số trang quản trị theo **Razor Pages** nếu nhóm muốn tận dụng tốc độ phát triển.

## 8.2. Repository interfaces đề xuất

```csharp
public interface IDocumentRepository
{
    Task<Document?> GetByIdAsync(Guid id);
    Task<PagedResult<Document>> GetPagedAsync(DocumentQuery query);
    Task AddAsync(Document document);
    Task UpdateAsync(Document document);
    Task DeleteAsync(Document document);
}

public interface IChatSessionRepository
{
    Task<ChatSession?> GetByIdAsync(Guid id);
    Task AddAsync(ChatSession session);
    Task<IReadOnlyList<ChatMessage>> GetMessagesAsync(Guid sessionId);
    Task AddMessageAsync(ChatMessage message);
}

public interface IExperimentRepository
{
    Task AddAsync(ExperimentRun run);
    Task<ExperimentRun?> GetByIdAsync(Guid id);
    Task<IReadOnlyList<ExperimentRun>> SearchAsync(ExperimentQuery query);
    Task UpdateAsync(ExperimentRun run);
}
```

## 9. Business Layer cần có những service nào?

## 9.1. DocumentService
Chức năng:
- Upload file
- Validate file
- Lưu metadata
- Enqueue indexing
- Re-index / delete document

## 9.2. FileProcessingService
Chức năng:
- Extract text từ PDF/DOCX/PPT/PPTX
- OCR fallback
- Chuẩn hóa text
- Trích metadata (trang, heading, slide)

## 9.3. ChunkingService
Chức năng:
- Fixed-size chunking
- Structure-based chunking
- Hybrid chunking

## 9.4. EmbeddingService
Chức năng:
- Gọi embedding API hoặc local model
- Cache embedding nếu cần
- Chuẩn hóa vector output

## 9.5. VectorStoreService
Chức năng:
- Upsert vector
- Search nearest chunks
- Delete theo document

## 9.6. ChatService
Chức năng:
- Nhận câu hỏi
- Lấy chat history
- Gọi retrieval
- Gọi LLM
- Tạo answer + citations
- Lưu conversation log

## 9.7. ExperimentService
Chức năng:
- Tạo experiment run
- Chạy benchmark theo test set
- Tính metric
- Lưu kết quả tổng hợp

## 9.8. CitationService
Chức năng:
- Sinh citation object từ chunk metadata
- Chuẩn hóa định dạng hiển thị citation

## 13. Gợi ý về background jobs

Dự án này có nhiều tác vụ dài, không nên xử lý trực tiếp trong request-response:
- Index tài liệu
- Re-index
- Chạy benchmark test set 50 câu hỏi
- OCR hoặc parsing file nặng

### Cách làm
- Dùng `BackgroundService`
- Hoặc dùng `Hangfire` nếu muốn dashboard job rõ ràng hơn

**Khuyến nghị cho đồ án:**
- Nếu muốn đơn giản: dùng `BackgroundService + Channel<T>`
- Nếu muốn trình bày đẹp và dễ demo: dùng `Hangfire`
